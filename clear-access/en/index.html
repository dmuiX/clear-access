What I set up

  • A single entry point for self-hosted tools

  • Clear rules for who can reach which tool
  • Same login for SaaS tools, when supported

A simple principle

Access is managed centrally. What happens inside each tool stays inside that tool.

Technical background (optional)

I place a central access layer in front of your tools. It decides who can reach which tool. Your applications remain unchanged — only the access point becomes consistent.

  • Self-hosted: one central domain with subdomains as the entry point
  • Login: your existing company login (Microsoft, Google, or on-prem)
  • SaaS: same login via standard SSO, if the tool supports it
  • Scope: I manage “who can reach which tool”; internal roles stay inside each tool
  • Optional: extra network protection so only known devices can reach the access point

No migration of users or data. The goal is a clear access state — not a rebuild.

Technical background (optional)

I place a small, central access layer in front of your tools. This layer controls who can reach which tool. Your applications themselves remain unchanged.

  • Self-hosted tools:
    Access runs through one central domain with subdomains. The tools do not need to be modified.
  • Login:
    Users sign in with your existing company account (Microsoft, Google, or on-prem).
  • SaaS tools:
    The same login can be used via standard SSO, as long as the tool supports it.
  • Access logic:
    I define who is allowed to reach which tool. Internal roles and permissions stay inside each application.
  • Optional network layer:
    Access can be limited to known devices only, before authentication even happens.

There is no migration of users or data. The goal is a clear access state — not a rebuild or long-term project.
